# Japanese Reading Practice — PWA

This is a static site, ready to publish via GitHub Pages, and installable
as a PWA (Progressive Web App) — visitors can "Add to Home Screen" and it
works offline after the first visit.

## How to publish

1. Create a new GitHub repository (it must be **public** for free GitHub
   Pages hosting).
2. Push the entire contents of this folder to the repo (everything
   should sit at the repo root, or inside `/docs` if you configure Pages
   that way).
3. In the repo's Settings → Pages, set the source to the branch/folder
   you pushed to.
4. GitHub will give you a URL like
   `https://<your-username>.github.io/<repo-name>/` — that's your site.

## Folder structure

```
index.html                  ← landing page, links to all 4 activities
restaurant_browse.html      ← pick a level, browse restaurant review pages
amazon_browse.html          ← pick a level, browse Amazon review pages
deck.html                   ← pick a level, swipe through dating profiles
match_game.html             ← pick activity + level, match 4 reviews to photos

manifest.json                ← PWA app manifest (name, icons, colors)
service-worker.js            ← enables offline support
icons/                       ← app icons

restaurant-reviews/
  Starter/
    manifest.json            ← list of pages for this level
    review_xyz.html          ← pages generated by render_restaurant_review_html()
  Elementary_1/
  Elementary_2/
  Pre-intermediate/
  Intermediate_1/

amazon-reviews/
  (same per-level structure as above)

dating-profiles/
  (same per-level structure as above, with pages from render_profile_html())
```

## Adding a new page

1. Run `main.py` as usual to generate a new review/profile HTML file.
2. Copy that HTML file into the matching `{type}/{level}/` folder.
3. Open that folder's `manifest.json` and add one line for your new
   file, e.g.:

   ```json
   [
     { "file": "review_20260601_101500.html", "title": "らーめん花月" },
     { "file": "review_20260615_093000.html", "title": "Your new restaurant" }
   ]
   ```

   The `title` is just what's shown in the list — pick anything readable.

4. Push the new files to GitHub. That's it — no rebuild step, no script
   to run. The browse/deck/match-game pages fetch `manifest.json` live,
   so the new entry shows up as soon as it's on GitHub.

## Why a manifest.json instead of automatic folder listing?

GitHub Pages serves static files only — there's no way for a webpage to
ask "what files are in this folder" without either (a) a small index
file like this manifest, or (b) querying GitHub's API at runtime, which
has a 60-requests-per-hour-per-visitor limit on public repos. Since this
site may be used by a group of people from the same network (e.g. a
school), the manifest approach was chosen to avoid that shared rate
limit ever becoming a problem.

## Notes

- The `restaurant-reviews/Starter/` and `dating-profiles/Starter/`
  folders include a couple of sample/test pages so you can see the
  structure working immediately. Feel free to delete them once you've
  added your own real pages — just remember to also remove their
  entries from that folder's `manifest.json`.
- Level folder names use underscores instead of spaces (`Elementary_1`,
  not `Elementary 1`) to avoid URL-encoding headaches — this is just the
  folder name on disk; the level labels shown in the UI still read
  naturally ("Elementary 1").
